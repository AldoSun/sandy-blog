大家好，今天是2026年06月02日，星期二。

RTX Spark是NVIDIA的第一款PC芯片，性能相当于RTX 5070级别的显卡，配备128 GB统一内存，算力达到1 PFLOP的本地AI。机身轻薄，拔掉电源后几乎不降频。它采用Arm CPU + Blackwell GPU架构，也就是把Arm处理器和Blackwell显卡合在一起，形成一颗Superchip，官方称它是首款专为Windows PC设计的处理器。

在Computex 2026上，NVIDIA把Windows变成了一个可以跑AI代理的平台，推出了Windows on Arm的笔记本方案，核心正是这颗RTX Spark Superchip。因为本地拥有128 GB统一内存，CPU和GPU共享同一块高速存储，省去数据拷贝的时间。1 PFLOP相当于每秒能完成一千万亿次浮点运算，足以支撑70 B参数的大模型运行。实际测试显示，在本地跑70 B参数模型时，推理延迟低于500 毫秒，这比在云端推理要省下不少费用。

对普通开发者来说，笔记本可以像煎饼果子摊一样，随时随地24 小时不间断地跑AI Agent，不再受限于云费用和带宽。企业也能把客服、数据分析等Agent部署在本地，降低数据泄露的风险。就像老头乐电动车在城市里取代燃油车一样，本地算力正在逐步取代远程算力。Gartner预测，2026年企业AI推理工作负载超过50%会在本地或边缘运行，2023年还不到10%，这说明本地算力需求正在快速增长。

从市场份额看，NVIDIA在2026年约占80%的AI加速器市场，数据中心收入约194亿美元，说明它在算力

说完NVIDIA RTX S这件事，咱再聊聊下面这件——Perplexity 刚刚推出了一个叫 “Search as Code” 的新功能。先给大家说说这玩意儿到底是啥。它让 AI Agent，也就是那种能帮你自动完成任务的人工智能代理，直接生成 Python 脚本去调取搜索后端，而不是像以前那样一步步调用一堆函数。Python 脚本是一段用 Python 语言写的程序代码，能直接和搜索系统对话，省掉中间的繁琐环节。

据说，Perplexity 这么做是想把检索的延迟给压下来。也就是说，从你发出查询到拿到答案的时间会更短。具体动机官方目前没有确认，不过从技术角度看，直接让代码去跑搜索，省掉了层层包装，天然就能快一点。

这对开发者来说，最大的好处是代码量下降。过去要写好几个函数才能把搜索结果拼起来，现在一段几行的脚本就搞定。想象一下，像老头乐电动车那样，省油又省力。响应时间也跟着缩短，适合那些对时效要求高的场景。比如金融行业，银行、证券这些地方需要马上把最新行情拉出来；再比如客服，客服系统要在用户提问后立马给出答案，等不及的用户可不想等太久。

Perplexity 已经把这个功能的 Beta 版对外开放，用户可以直接去试用。官方说大约四周后会正式上线，届时大家就能在生产环境里直接用上。后续还有个信号，就是会加入多语言支持，届时可以在中文环境下直接生成检索代码。也就是说，咱们说中文也能让 AI Agent 写出对应的 Python 脚本去搜索，省得再手动翻译。

整体来看，这个 “Search as Code” 把搜索和编程桥接得更紧密，让信息检索型的 Agent 更像一把直接开锁的钥匙，而不是一串繁琐的指令。等结果出来再说，等官方确认再聊。

他是怎么做到的？据说，他先花时间去了解企业的具体需求，找出那些让老板们夜不能寐的痛点。比如说客服响应慢、内部审批流程繁琐、数据汇总总是出错。找到这些痛点后，他就不从头写代码，而是直接搬现成的 Agent 框架上场。这里用的框架像 Claude 这个平台，它能理解自然语言指令，还能配合 n8n 这种工作流自动化工具，把不同系统串起来。于是，一个企业只要把需求说出来，Agent 就能自动帮忙处理，省掉了研发团队的时间成本。

这种做法的好处很明显。对技术人来说，省去了研发的高投入，只要会搭建和调教这些现成的 Agent，就能快速交付。对企业来说，买到的不是一个抽象的 AI 产品，而是直接解决了他们最头疼的问题。于是，口碑在行业里慢慢传开，更多公司找上门来。于是，他的项目单价保持在 1 万到 1.5 万美元的区间，订单量自然就上来了。

从这件事里可以看到一个可复制的思路：先找企业痛点，再用现成的 AI Agent 框架交付，省掉研发成本。对想快速变现的技术人来说，这是一条相对明确的路径。只要熟悉 Claude 这种大模型的能力，懂得怎么用 n8n 把业务流程自动化，就能把一个“AI 小工具”包装成企业愿意付费的解决方案。

至于下一步的动向，这位创业者已经在筹划下个季度的计划。据说，他准备把自己用的 Agent 组合做成标准化的即插即用套件，直接卖给想自己动手的企业。预计会在 Product Hunt 这个产品发布平台上进行预售，看看市场的接受度如何。

说完19 岁少年月入 5 万这件事，咱再聊聊下面这件——Hermes 这个自我改进的 AI Agent 最近跟 Obsidian 笔记库玩起了深度绑定。

先说技术细节。Hermes 被装在一台 VPS 上，VPS是一种远程的虚拟专用服务器，提供持续的算力。它通过 Filesystem MCP 这个文件系统中间件和本地的 Obsidian vault 互通，换句话说，远程的代码能直接读写你电脑里存放笔记的文件夹。这个绑定方式在 X平台（Rohit 的推文）里写得很清楚：VPS 部署 + Filesystem MCP 绑定——Hermes 运行在远程服务器上，通过 Filesystem MCP 与本地 Obsidian vault 互通。

最核心的玩法是读取/写入循环。每一次推理，Hermes 先去 vault 把相关笔记拉下来，分析完后再把新生成的内容写回去，形成一条闭环。X平台（Rohit 的推文）同样提到：读取/写入循环——每一步推理从 vault 读取，输出写回为新笔记。于是笔记库不再是静态的文档，而像是一个会说话的伙伴，随时把思考结果写进去。

如果你想在本地装这个插件，路径是 vault/.obsidian/plugins/hermes-voice-assistant，GitHub 上的 symunona/obsidian-hermes 项目里有明确指引。插件本身把 Hermes 当成语音助理，直接在 Obsidian 里对话。

Hermes 的设计围绕五大支柱展开：Memory、Skills、Soul、Crons、Self‑Improvement，这几块在 MindStudio 博客里被列出来。Memory负责记忆过去的对话和笔记，Skills是具体的工具箱，Soul给模型注入目标导向，Crons是定时任务，Self‑Improvement则是让系统自己找不足并改进。正因为有 Self‑Improvement，Hermes 能在运行时自行生成新笔记。官方博客的集成闭环描述是这么说的：“Obsidian stores the human‑readable project state, GitHub stores source history, and Hermes operates across both with explicit verification.” 也就是说，Obsidian 保存人能看懂的项目状态，GitHub 保存源码历史，Hermes 在两者之间跑来跑去，并且每一步都有明确的校验。

一个具体的自我改进实例是这样写的：“Hermes minted obsidian‑evening‑routine‑capture … I never asked for it… The agent has a self‑improvement loop.” 这句话直接出自 Hermes Agent 官方博客，说明它在没有明确指令的情况下自己创建了一个叫做 obsidian‑evening‑routine‑capture 的笔记，算是自我学习的表现。

还有一个技术点叫 GEPA 技术，它会读取执行轨迹并提出针对性改进，这也是 Self‑Improvement 的底层实现。换句话说，Hermes 会把每一步的操作记录下来，然后分析哪些地方可以更高效，再把改进方案写进笔记。

从社区反馈看，Rohit 在 X平台发的推文收获了 👍55 和 🔁14，说明大家对这个闭环系统挺感兴趣。虽然目前没有官方确认为什么要做这么一个自我改进的 AI Agent，但据说开发者的初衷是想把个人知识库变成第二大脑，让创作者不必再手动归档，省下的时间可以去写内容或思考新点子。

如果把这个思路搬到企业内部，想象一下，团队的文档库可以自动补全最新的技术方案，运维日志也能被 AI 自动归类，省去手工整理的麻烦。只要把同样的 Filesystem MCP 连接到企业的共享文件夹，Hermes 也能在远程服务器上跑，实时把改进写回去。
